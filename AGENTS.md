# AGENTS.md

## Project Overview

Automoto Report is an interactive one-page reporting tool for Instagram creator and content campaign activity. The main goal of the project is to help clients review campaign performance and provide clear instructions for future content creation campaigns.

The product should remain practical, client-facing, and easy to use. Prioritize clarity, fast review of campaign data, and simple workflows for entering or preparing campaign guidance.

## Current Stack

- Static frontend: `index.html`, `styles.css`, and `main.js`.
- Local development server: `server.js`, using only built-in Node.js modules.
- Package entry point: `package.json`.
- Data source: JSON files in `data/`.
- No build step and no required dependency installation.
- XLS/XLSX import uses SheetJS from a CDN in `index.html`; CSV import runs in the browser.

Run locally with:

```bash
npm run dev
```

The default local URL is:

```text
http://localhost:5173
```

## Core Files

- `index.html`: Main app shell and external browser libraries.
- `styles.css`: All application styling and responsive layout.
- `main.js`: Frontend state, authentication fallback, routing, rendering, charts, import handling, and UI interactions.
- `server.js`: Local Node server, session cookies, password login, protected static assets, and share-link creation.
- `data/report-data.json`: Main report content and campaign performance data.
- `data/promoted-models.json`: Reference list for promoted model detection from captions, URLs, hashtags, and variants.
- `data/share-links.json`: Generated by the server when admin users create share links.

## Product Goal

The app exists to support content creation campaign planning and reporting for clients. Future changes should help clients:

- Understand creator activity and campaign performance by period.
- Review content formats, themes, reach, impressions, engagements, spend, and related KPIs.
- Compare brand and competitor activity where relevant.
- Share report views safely with stakeholders.
- Provide actionable instructions for upcoming content creation campaigns.

## Authentication And Sharing

The local Node server supports two roles:

- Admin: can access report functionality intended for internal use, including import preview and share-link creation.
- Viewer: can review the report without admin-only actions.

Default passwords are documented in `README.md` and can be overridden with environment variables:

```bash
ADMIN_PASSWORD="..." VIEWER_PASSWORD="..." npm run dev
```

Static hosting can only provide a client-side access gate. Real data protection requires the Node server or another backend-capable hosting environment.

Admin users can create view-only share URLs. Generated tokens are stored in `data/share-links.json`.

## Development Guidelines

- Keep the app lightweight: do not introduce a framework or build pipeline unless the task clearly requires it.
- Prefer plain JavaScript, semantic HTML, and scoped CSS that follows the existing structure.
- Preserve relative paths so the app continues to work on GitHub Pages and project URLs.
- Treat `data/report-data.json` as the source of truth for report content.
- Keep client-facing copy clear, professional, and concise.
- Avoid unrelated refactors when making focused changes.
- Maintain the admin/viewer distinction when adding features.
- Do not expose admin-only tools to viewer or share-link sessions.
- Be careful with credentials, session behavior, and share-link handling.

## Data Notes

The report data is organized around periods and creators. Important fields include:

- `activePeriodId`: the default period shown on load.
- `periods`: available reporting periods.
- `creators`: creator rows for each period.

When changing data shape, update the rendering logic in `main.js` and keep existing report views working.

## UX Priorities

- The first screen should remain the usable report experience, not a marketing page.
- Campaign insights should be scannable for busy clients.
- Controls should be obvious and role-appropriate.
- Responsive behavior matters; verify both desktop and mobile layouts after visual changes.
- Use clear labels for content campaign planning, creator performance, campaign periods, and share/report actions.
